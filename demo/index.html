Tier caps & how the hierarchy works

Caps (all deliberate, nothing hidden): GPU 47 GB is hardware truth (80 GB − 28 GB weights, ~23 docs). CPU tier: 60 GB pinned host RAM by config (~30 docs). NIXL pool: 1,700 slots × 48 MiB/chunk (256 tok × 192 KiB) = 85.6 GB on NVMe (~41 docs). A pyramid — each tier bigger than the one above.

Every doc is ~10k tokens (~30 pages) costing ~2 GB of KV (0.19 MB/token). KV is written through to all tiers; a full tier LRU-evicts its oldest while lower tiers still hold them. Docs A–E are docs 1–5 of set 1. TTFT decoder: ~1.8s cold prefill · ~0.07s GPU · ~0.17s CPU RAM · ~0.35s NIXL/NVMe.

Suggested experiment (you drive every step)

① Sweep docs 1–10 — cold, ~1.8s each; every tier gauge fills by 20 GB. ② Sweep 1–10 again — all GPU hits, ~0.07s. ③ Sweep 11–20, 21–30, 31–40 — 80 GB floods the pyramid; GPU pins at 100%, then CPU. ④ Sweep 1–10 again — GPU evicted them long ago, but a lower tier still has them: fast, no prefill. ⑤ Clear CPU tier → sweep 11–20 — served from NIXL/NVMe (~0.35s), still no prefill. ⑥ Clear all three tiers → any sweep = full recompute. That's baseline life, permanently.

How the tier evidence works

Every click re-runs real, read-only commands on the GPU server. GPU tier: nvidia-smi, the KV-budget log line, live vLLM cache metrics. CPU/NIXL tiers: evidence_lookup.py tokenizes each demo doc with the chat template and asks the LMCache controller's /lookup which tier holds those tokens — totals come from LMCache's own index and match the gauges.

Fine print: audit covers the 40 demo docs; /lookup names each doc's fastest tier. Blind spot (LMCache 0.5.1): the NIXL backend never reports to the controller registry, so pool-only docs are invisible to the audit — after a CPU clear it reads empty while the pool still serves (~0.35s + a "Retrieved" log line proves it). The storage gauge models true pool membership.
