Tier caps & how the hierarchy works

Caps (all deliberate, nothing hidden): GPU 47 GB is hardware truth (80 GB − 28 GB weights, ~23 docs). CPU tier: 60 GB pinned host RAM by config (~30 docs). NIXL pool: 1,700 slots × 48 MiB/chunk (256 tok × 192 KiB) = 85.6 GB on NVMe (~41 docs). A pyramid — each tier bigger than the one above.

Every doc is ~10k tokens (~30 pages) costing ~2 GB of KV (0.19 MB/token). KV is written through to all tiers; a full tier LRU-evicts its oldest while lower tiers still hold them. Docs A–E are docs 1–5 of set 1. TTFT decoder: ~1.8s cold prefill · ~0.07s GPU · ~0.17s CPU RAM · ~0.35s NIXL/NVMe.

Suggested experiment (you drive every step)

① Sweep docs 1–10 — cold, ~1.8s each; every tier gauge fills by 20 GB. ② Sweep 1–10 again — all GPU hits, ~0.07s. ③ Sweep 11–20, 21–30, 31–40 — 80 GB floods the pyramid; GPU pins at 100%, then CPU. ④ Sweep 1–10 again — GPU evicted them long ago, but a lower tier still has them: fast, no prefill. ⑤ Clear CPU tier → sweep 11–20 — served from NIXL/NVMe (~0.35s), still no prefill. ⑥ Clear all three tiers → any sweep = full recompute. That's baseline life, permanently.

How the tier evidence works (what runs when you click a tier card)

Every click re-runs real commands on the GPU server and shows their raw output — nothing is cached or simulated, and nothing is modified (all commands are read-only). The panel opens with the dashboard's own gauge value so you can compare it against the system's answer below.

GPU tier: nvidia-smi, the vLLM log line stating the KV budget, and vLLM's live cache metrics. CPU RAM and NIXL tiers: a small audit script (evidence_lookup.py) that, for each of the 40 demo docs, ① rebuilds the doc's exact prompt, ② tokenizes it through vLLM's /tokenize with the chat template applied (the cache is keyed on the templated token sequence — raw text tokens never match), and ③ asks the LMCache controller's /lookup API which tier holds those tokens. The per-doc rows and GB totals come from LMCache's own index, not from this dashboard — when they match the gauge, that's two independent sources agreeing.

Fine print: the audit covers the 40 demo docs (cold-prefill and chat prompts aren't in the corpus, so they can't be looked up); /lookup names each doc's fastest tier — a doc in CPU RAM usually also has a copy in the NIXL pool below. Known blind spot: in LMCache 0.5.1 the NIXL backend never reports to the controller registry, so docs held only in the pool are invisible to the audit (after a CPU clear it reads empty while the pool still serves — prove it by clearing GPU+CPU and asking a swept doc: ~0.35s and a "Retrieved" log line). The storage gauge models true pool membership. Two deployment requirements made this work, both relevant in production: lookups must use chat-template tokens, and every LMCache process (engine + controller) must run with the same PYTHONHASHSEED or the index keys never match.
