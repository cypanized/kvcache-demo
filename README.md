# KV Cache Tiering Demo — vLLM + LMCache + NVIDIA NIXL

Interactive demo and benchmark showing KV-cache tiering from GPU HBM → pinned CPU RAM →
NVMe via NVIDIA NIXL, on a rented vast.ai A100. Repeat-request TTFT drops up to **10×**
when a tier catches what the GPU evicts.

**Measured tier ladder** (Qwen2.5-14B, A100 80GB):
cold prefill **1.8s** → GPU hit **0.07s** → CPU RAM **0.17s** → NIXL/NVMe **0.35s**.

## Quick start

```bash
pip install --user vastai && vastai set api-key <KEY>
vastai create ssh-key "$(cat ~/.ssh/id_ed25519.pub)"
./deploy/deploy.sh        # rents an A100, deploys everything, ~5-8 min
# → dashboard at http://127.0.0.1:7811 · tear down: vastai destroy instance <ID>
```

## The demo (dashboard)

Two serving modes — **Baseline** (GPU only: eviction = amnesia) and **Full hierarchy**
(GPU 47 GB → CPU 60 GB → NIXL pool 85.6 GB, write-through pyramid). Manual 10-doc
sweeps, instant per-tier clear buttons with dropped-token receipts, live residency
gauges with stated caps, click-for-evidence terminals (real commands + LMCache
controller `/lookup` audit), free-form prompt box, persistent per-request TTFT log.

## Hard-won gotchas (all handled by the deploy kit, documented in the guide)

- `nixl_pool_size` is a slot **count** (× 48 MiB/chunk), not bytes — bytes OOM-kills the engine
- POSIX NIXL backend requires `nixl_buffer_device: cpu` and a raised `ulimit -n`
- `PYTHONHASHSEED` must be pinned in **every** LMCache process or controller `/lookup`/`/clear` silently match nothing
- Cache lookups need **chat-template** tokens, not raw prompt tokens
- LMCache 0.5.1's NIXL backend lacks `clear()` (patched in `bootstrap.sh`) and never reports
  to the controller registry (documented blind spot)
- `use_direct_io: true` deadlocks 0.5.1 at startup
- vLLM engine cores rename to `VLLM::EngineCore` — naive pkills orphan them holding GPU memory
- fresh vast.ai instances sometimes boot to `exited` — auto-started by `deploy.sh`
